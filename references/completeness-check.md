# 逻辑完整性检查清单

## 核心原则

**产品/技术文档必须保证逻辑链条完整**。任何"逻辑链上的关键环节"都不能省略——即使源 Markdown 没写，也要主动补上。

判定标准很简单：**这一节删了，逻辑链会不会断？** 如果会断，就是核心环节，必须独立成章。

## 通用检查（所有文档类型）

任何技术/产品文档都该问的 5 个问题：

1. **数据从哪来？**（输入）
2. **错误如何处理？**（异常路径）
3. **如何回滚 / 失败恢复？**（容错）
4. **如何监控 / 告警？**（可观测性）
5. **决策依据是什么？**（为什么这么做）

如果文档对这些都没回答，就是逻辑断链。

## 按文档类型的完整性清单

### 产品设计文档

| 章节 | 容易遗漏的关键环节 |
|---|---|
| 产品定位 | ❌ **目标用户的排除理由**（为什么不是另一群人） |
| 价值主张 | ❌ **与竞品的根本差异**（不是功能差异，是机制差异） |
| 产品形态 | ❌ **平台约束**（如 WhatsApp 24h 窗口、Apple IAP 抽成） |
| 核心场景 | ❌ **异常场景**（用户拒绝、券失效、网络断） |
| 信息架构 | ❌ **数据来源与供给体系** ⭐（这是产品逻辑核心，常被当"技术细节"省略） |
| 交互设计 | ❌ **跨渠道/跨端一致性** |
| 信任设计 | ❌ **决策溯源**（为什么必做/红线） |
| 留存机制 | ❌ **数据飞轮**（怎么越用越准） |
| 成功指标 | ❌ **反指标红线**（什么情况下立即审计） |
| 商业模式 | ❌ **变现对象**（向谁收费，与谁利益冲突） |
| 风险与红线 | ❌ **未决项**（哪些 Phase 1 不解决，但要知道） |

**最常遗漏**：数据来源。表现为：
- 写了"反馈循环回写 validity_status 字段"，但没说"这些数据怎么进库"
- 写了"红线不做服务端代注册"，但没说"那做什么"
- 写了"拉美裔商家联盟覆盖不足"，但没说"联盟是数据来源主干"

### 技术方案文档

| 章节 | 容易遗漏的关键环节 |
|---|---|
| 系统架构 | ❌ **失败场景**（单点故障、级联失败） |
| 数据模型 | ❌ **数据生命周期**（创建/更新/删除/归档） |
| API 设计 | ❌ **错误码体系** + **限流策略** + **鉴权** |
| 性能 | ❌ **压测基准**（QPS 目标、P99 延迟） |
| 部署 | ❌ **回滚策略** + **灰度方案** |
| 监控 | ❌ **告警阈值** + **oncall 升级路径** |
| 安全 | ❌ **PII 处理** + **审计日志** |

### API 文档

| 章节 | 容易遗漏的关键环节 |
|---|---|
| 接口定义 | ❌ **错误响应**（不只是成功路径） |
| 鉴权 | ❌ **Token 过期** + **刷新机制** |
| 限流 | ❌ **429 响应** + **Retry-After** |
| 分页 | ❌ **游标 vs 偏移量** + **总数预估** |
| 字段 | ❌ **必填 vs 可选** + **默认值** + **取值范围** |
| 版本 | ❌ **Breaking change 政策** |

### 操作手册 / Runbook

| 章节 | 容易遗漏的关键环节 |
|---|---|
| 前置条件 | ❌ **权限要求** + **环境变量** |
| 步骤 | ❌ **回滚步骤**（每步失败怎么办） |
| 验证 | ❌ **成功标志**（怎么确认操作完成） |
| 故障排查 | ❌ **常见错误** + **诊断命令** |
| 联系 | ❌ **oncall 升级路径** |

## 主动补全的工作流

当发现源 Markdown 缺了关键环节：

### Step 1 · 标记为"基于已有素材补充"

不是凭空捏造。先检查：
- 同一项目的其他文档是否有讨论？（如这次数据来源在《Coupon-Giveaway聚合Agent_技术可行性与成本分析.md》里有详尽讨论）
- 行业常识/标准做法是什么？
- 是否需要问用户？

### Step 2 · 章节内注明依据

补的内容要标明依据，避免被误以为是凭空创作：

```markdown
> 按《XXX 文档》结论：...
```

或

```markdown
（基于通用最佳实践补充，源 Markdown 未展开）
```

### Step 3 · 同步更新源文档

衍生 HTML 补了，源 Markdown 也要同步。否则会出现"哪份是真相"的混乱。

## 这次真实案例（本次 skill 创建的源起）

**场景**：用户给了 v4 HTML 产品设计文档，我重构完。

**用户反馈**："产品文档还没有对优惠券的数据来源进行讨论，产品逻辑的成立少了最关键的一环！"

**我当时的问题**：写 v4 时把"数据来源"误判为"技术细节"省略了。但实际上：
- 6.1 反馈循环写 `validity_status` 字段回写 → **这字段怎么填充？** ← 数据来源
- 8.4/15.2 红线"不做服务端代注册" → **那做什么？** ← 数据来源
- 15.1 风险 #3 "拉美裔联盟覆盖不足" → **联盟是什么？** ← 数据来源
- 3.1 价值主张"决策确定性" → **怎么保证？** ← 有效性校验（数据来源子节）

**正确处理**：
1. 承认遗漏
2. 新增章节 5：数据来源与供给体系（6 个子节）
3. 原 5-14 重编号为 6-15
4. 同步更新 markdown 源文档

## 教训沉淀

判断"是否省略"的**反向问法**：

- ❌ 旧："这节内容能不能折叠让排版更简洁？"（容易把核心当细节省略）
- ✅ 新："这节内容是核心架构/决策依据/逻辑链环节吗？"

如果是 → 展开 + 视觉强化。
如果否 → 才考虑折叠。

**判断标准**：如果某内容是产品逻辑链条上的关键环节（删了之后逻辑断链），它就是产品设计的核心，不是技术细节。

## 检查清单（写完 HTML 后过一遍）

- [ ] 通读全文，画出逻辑链
- [ ] 每个逻辑链环节都有对应章节吗？
- [ ] 有没有"逻辑链断点"（前文提到的概念，后文没解释来源/依据）？
- [ ] 红线/必做项有依据吗？（不能只是断言）
- [ ] 风险/未决项有上下文吗？（不能孤立存在）
- [ ] 反馈/循环机制有完整闭环吗？（输入→处理→输出→反馈→输入）
- [ ] 数据来源明确吗？
- [ ] 错误/异常路径有覆盖吗？
- [ ] 多载体同步（markdown + HTML 都更新）？
