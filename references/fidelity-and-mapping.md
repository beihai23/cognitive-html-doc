# 转换模式 + 内容保真门 + 源章节映射

本文是 SKILL.md **Step 1.5（转换模式 + 内容保真门）** 和 **Step 5.5（源章节映射 + 交付报告）** 的展开。核心解决一个问题：**如何让"认知降维"不退化成"信息丢失"**。

## 核心原则

> 认知降维 = 降低阅读成本，不是降低信息完整度。

降维重组的是**呈现方式**（空间位置 / 视觉权重 / 交互组件），不是删减信息量。同一份规格，可以重组得更易扫读，但不该因为它"长"就摘要掉。

---

## 一、转换模式（3 选 1，默认双层型）

写 HTML 前必须先选模式。模式决定"允许对已有内容做多大改动"。

| 模式 | 适用场景 | 对已有内容的处理 | 压缩预算（启发式） |
|---|---|---|---|
| **保真型** | 正式 PRD、研发规格、合同、操作手册 | 逐章保留，只重组呈现，不摘要 | ~95%（几乎不压缩，只去真重复） |
| **双层型（默认推荐）** | 大多数产品/技术文档 | 主线精简 + 完整规格折叠保留 | ~85%（主线可收敛，规格层保留全文） |
| **摘要型** | 汇报、路演、管理层快速评审 | 允许大幅压缩 | 由用户显式定（如"控制在 N 页"） |

### 判定流程

```
这份文档的读者是谁、拿来做什么？
├── 要被研发/运营拿去落地（含字段/状态/异常/验收）
│   └── 用户没明说 → 默认【双层型】
│       用户要求"完整规格书" → 【保真型】
├── 给高管/评审快速看（不需要落地）
│   └── 用户明说"汇报/路演/控制页数" → 【摘要型】
└── 不确定 → 默认【双层型】，并在交付时说明所选模式
```

**关键约束**：不能由模型自行默认摘要型。摘要型是用户**显式要求**"给高管看 / 控制 N 页 / 路演"时才用。一份要落地的规格被悄悄摘要掉字段，是这个 skill 最典型的失败。

### 为什么默认双层型

它同时满足两类读者：
- **决策者**（3 分钟扫主线）：Hero + 架构图 + 10 个左右主线章节 → 快速理解产品
- **落地者**（查字段/状态/异常）：每章下挂折叠规格层 → 完整保留，按需展开

这正是"双层"的含义：**主线层做认知降维，规格层做信息保真**。两层各司其职，互不牺牲。

## 二、内容保真门（6 类标签）

选完模式后、动手设计 HTML 前，把每个源章节归到下面 6 类之一。这一步决定"这段内容允许怎么处理"。

| 标签 | 处理方式 | 例子 | 能不能删？ |
|---|---|---|---|
| **核心需求** | 100% 保留，展开 + 视觉强化 | 产品定位、核心场景、MVP 必做项 | ❌ 绝不 |
| **决策依据** | 保留（可移位到决策点旁），不摘要 | "为什么选这群人"的 5 个理由、技术选型对比 | ❌ 绝不 |
| **执行规格** | 保留（可折叠），**折叠保留全文不摘要** | 字段定义、状态机、错误码、验收标准、配置项 | ❌ 绝不（可折叠但保全文） |
| **重复表达** | 合并到唯一主定义处，其他位置锚点引用 | "产品定位"在概览和详述里各完整写一遍 | ⚠️ 合并（不是删除） |
| **历史说明** | 进附录 | 变更日志、已废弃方案、历史决策 | ✅ 移入附录 |
| **可删除** | **必须说明删除理由**，否则不删 | 与主题无关的离题内容、重复元信息 | ✅ 须写理由 |

### 关键区分：去重 ≠ 删减

**去重 = 合并重复表达，不是删减不同层次的细节。** 这是最容易踩的坑。

源文档常有这样的结构：
- §1 概览讲了产品定位
- §8 产品详述又完整讲了一遍定位

这是**语义重复**（同一结论给同一类读者），应该合并——保留 §8 完整定义，§1 改成锚点"详见 §8"。

但下面这些**不是重复**，是**必要重复**，都该保留：

| 看似重复的两组 | 为什么是必要重复（都保留） |
|---|---|
| 概览 vs 详细规格 | 不同读者、不同目的（决策者扫概览 / 落地者查详述） |
| 主流程 vs 数据模型 | 回答不同问题（怎么跑 / 数据长什么样） |
| MVP 范围 vs 验收标准 | 回答不同问题（做什么 / 怎么算做完） |
| 风险 vs 对应产品机制 | 互为参照（风险点 + 该风险的缓解机制） |

**一句区分法**：问"这两处是给**同一类读者、回答同一个问题**吗？"——是 → 合并；否（不同读者/不同问题）→ 都保留，可用锚点交叉引用。

详见 SKILL.md 反模式 #14。

## 三、压缩预算（启发式，非硬约束）

不同模式有不同的"允许压缩多少"参考线。**这些是默认值/启发式，不是绝对阈值**——用户有明确要求时以用户为准，但模型自行压缩时应守住这条线，压缩明显超出时要主动提示。

| 模式 | 建议有效内容保留率 | 硬性不丢的内容 |
|---|---|---|
| 保真型 | ~95% | 全部 |
| 双层型 | ~85% | 核心需求 / 决策依据 / 执行规格 = **100%**；允许收敛的是重复表达和历史说明 |
| 摘要型 | 由用户定 | 核心需求 + 决策依据 = 100%；执行规格可摘要 |

**几个具体信号**（出现说明可能压缩过度）：
- 把字段定义表摘要成一句"包含若干字段" → 过度
- 把状态机摘要成"有几个状态"而不保留转换条件 → 过度
- 把错误码列表删掉只留"有完整错误处理" → 过度
- 把验收标准省略只留"有明确验收" → 过度
- 主线章节合并后，源文档里的具体数字/阈值/版本号消失 → 过度

**"避免重复" ≠ "减少信息量"**：用户说"避免重复"，指的是**合并语义重复**，不是"少写点"。不要把后者当前者执行。

## 四、源章节映射表（Step 5.5 用）

交付 HTML 前，生成一张 `源章节 → HTML 章节 → 处理方式` 的映射。目的是**自证没有无意遗漏**——每一行源章节都必须出现在表里。

### 模板

| # | 源 Markdown 章节 | 对应 HTML 位置 | 处理方式 | 备注 |
|---|---|---|---|---|
| 1 | §1 定位 | Hero 区 | 完整展开 | — |
| 2 | §2 目标用户 | §1 主线 | 完整展开 | 决策依据保留 |
| 3 | §3 用户画像细节 | §1 折叠 | 折叠保留 | 落地细节 |
| 4-6 | §4-6 场景/模式/商业 | §2 主线 | 合并呈现 | 收敛为主题章 |
| 7 | §7 数据来源 | §3 | 完整展开 | 核心环节 |
| 8-10 | §8-10 字段/状态/异常 | §4 折叠规格层 | 折叠保留 | 全文保留 |
| 11 | §11 变更日志 | 附录 A | 附录保留 | — |
| — | （无） | — | — | 无删除 |

**处理方式只能是 5 种**：完整展开 / 合并呈现 / 折叠保留 / 附录保留 / 删除并说明理由。

**自检**：源文档有 N 章，映射表必须有 N 行（合并的多章算多行，都指向同一 HTML 位置）。源有 21 章但表里只有 11 行 = 10 章下落不明 = 可能丢内容。

## 五、交付报告（精简版，5 行）

交付 HTML 时附一份简短报告，让用户一眼看到"我动了什么"。不需要长篇，5 行足够：

```
模式：双层型 | 源 21 章 → HTML 11 节
合并：§4-6 合为主题章"产品主线"（详见映射表）
删除：无 / 或 删了 §X（理由：与主题无关）
折叠保留的核心规格：6 处（字段定义 / 状态机 / 错误码 / 验收标准 / 配置项 / 异常处理）
估算保留率：~90%（核心需求 100%，规格层全文保留）
```

**为什么必须附报告**：双层型/保真型的产出乍看比源文档短很多（主线收敛了），用户会担心"是不是丢了内容"。报告把"短"的原因说清楚（是合并/折叠，不是删除），信任就建立了。

## 六、真实案例对照（v15 vs v16）

本 skill 的这一套机制，源自一次真实的对照复盘：同一份源 Markdown，两个执行方产出差异巨大。

| 维度 | 保真型执行（v15） | 摘要型执行（v16） |
|---|---|---|
| 文件大小 | 109 KB | 48 KB |
| 可见文本 | ~15,000 字符 | ~7,400 字符 |
| 章节数 | 25 | 11 |
| 卡片数 | 121 | 37 |
| 图表数 | 8 | 5 |
| 折叠区域 | 0 | 6 |

### 各自的优缺点

**v15（保真型）值得借鉴**：
- 完整保留目标用户、用户故事、采用路径、商业模式、待确认决策等产品信息
- 产品/研发/数据/运营都能查到具体规格
- 与源 Markdown 章节对应，容易审查遗漏
- 关键内容全部展开，不因折叠被忽略

**v15 的问题**：
- 24 个导航项认知负担高
- 121 张卡片导致视觉重点稀释
- 产品定位/双引擎/流程存在语义重复
- 更像"漂亮的完整规格书"，不够适合快速决策

**v16（摘要型）值得借鉴**：
- 主线从 21 章收敛为 10 个主题，首屏效率高
- 归因型 Campaign、自动 Verify、Trace 等成为视觉重点
- 卡片和图表使用克制

**v16 的问题**：
- 页面版本、报告字段、SDK 诊断、数据保留、具体指标被过度摘要
- 缺少完整的需求查阅层
- 更适合汇报和评审，**不足以独立支撑研发落地**

### 教训

- **v15 内容保真更好，v16 信息架构更好。**
- 理想版本 = **以 v16 为骨架（主线收敛），以完整 Markdown 内容为血肉（规格保真）**。
- 这正是"双层型"模式 + "内容保真门"要解决的问题：主线层学 v16，规格层学 v15。
