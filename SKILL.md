# Cognitive HTML Doc

把密集的 Markdown 重构为**认知降维**的工业级 HTML 文档。

## 核心哲学

Markdown 的本质是纯文本，长篇阅读认知负荷极高。这个 skill 的目标是**打破平铺直叙**，利用 HTML 的空间布局、交互组件、图形化能力，把信息降维成"扫视即可获取"的结构。

**最关键的一条约束（所有后续原则的纲领）**：

> **认知降维 = 降低阅读成本，不是降低信息完整度。**
> 降维是重组信息的**呈现方式**（空间位置 / 视觉权重 / 交互组件），不是删减信息量。同一份规格，可以重组得更易扫读，但不该因为它"长"就摘要掉。

为什么单独拎出来：本 skill 曾被误用成"编辑重构型摘要器"——把 13,000 字符的产品规格压成 7,400 字符交付，丢失了字段、状态、异常、验收标准等研发落地必需的细节。**根因不是某个步骤写错了，而是全文没设信息完整度的下限**。所以这条约束要放在最前面，后续的"转换模式 / 内容保真门 / 源章节映射"都是为了把这条落实。

读者阅读技术文档时分三种节奏，产出必须同时满足：

- **3 秒抓核心**：Hero 区一句话定位 + 4 维度卡片 → 整个产品/系统是什么、给谁、怎么实现、衡量什么
- **30 秒理解全貌**：主架构图（颜色编码）→ 系统拓扑一目了然
- **3 分钟查细节**：固定侧边栏 TOC + 滚动高亮 → 直达目标章节

## 适用文档类型 + 识别场景锚定

这个 skill 的所有原则都**场景无关**——适用于产品文档、技术方案、API 文档、操作手册、算法说明、研究报告等任意"密集文字需要降维"的场景。

**容易踩的坑**：本 skill 的例子大多是产品文档的（北极星指标、用户画像、Phase 路径），但**原则本身就是原则**，不要因为例子偏某个场景就误以为"原则只在那个场景适用"。

### 如何识别"通用原则被场景词锚定"

写每个原则时，问自己 3 个问题：

1. **这个原则只适用于 [产品/技术/API] 文档吗？** → 如果不是，它是通用原则
2. **我用的词是否暗示了特定场景？**（"KPI" 暗示产品；"SLO" 暗示技术；"限流值" 暗示 API）
3. **我的例子是否只来自一个场景？** → 如果是，加其他场景例子

### 抽象到场景无关的方法

| 错误（锚定） | 正确（场景无关） |
|---|---|
| "KPI 应该突出" | "关键数字应该突出（产品 KPI / 技术 SLO / API 限流值）" |
| "护城河要前置" | "核心价值要前置（产品护城河 / 技术核心机制 / API 差异化能力）" |
| "5 个理由要展开" | "决策依据要展开（产品选用户群的'为什么'/ 技术选型的'为什么'）" |
| "Phase 1 必做项" | "MVP 必做项（产品 Phase 1 / 技术 v1.0 / API 稳定接口）" |

**核心原则**：**通用原则用中性词，多场景配例子，场景词只下沉到具体例子里**。

## 何时触发

| 触发场景 | 例子 |
|---|---|
| 明确要求 HTML 化 | "把这份 markdown 转成 HTML"、"做一份工业级 HTML 文档" |
| 要求视觉重构 | "这个 markdown 太长了，能可视化吗"、"降维成扫视即可获取的结构" |
| 提供蓝图 + 期望高级呈现 | 用户给出 markdown + 提示词要求"高可读性、零依赖、单文件" |
| 长文档工程化 | 文档超过 500 行，需要 TOC、章节锚点、状态色系统 |

**不要用于**：一行命令就能搞定的 markdown 渲染（如 `pandoc`）；纯打印 PDF；不需要交互的简单文档。

## 工作流程

5 个主步骤（Step 1–5）保留原有编号，中间插入两个**门控步骤**——它们不取代任何现有步骤，只补上"信息完整度"这条原本缺失的轴：

- **Step 1** 通读画逻辑链
- **Step 1.5** ⭐ 选择转换模式 + 内容保真门（新）
- **Step 2** 检查逻辑完整性
- **Step 3** 设计信息分层
- **Step 4** 选表现形式（含视觉节制）
- **Step 5** 系统扫描每章
- **Step 5.5** ⭐ 源章节映射 + 交付报告（新）

### Step 1 · 通读 Markdown，画逻辑链

不要直接开始写 HTML。先通读全文，画出**核心逻辑链**：

```
输入（数据/触发） → 处理 → 输出 → 反馈/沉淀
```

这一步决定后续每节内容是"逻辑链上的哪一环"。**逻辑链断裂的章节就是要补的关键环节**（见 Step 2）。

### Step 1.5 · 选择转换模式 + 内容保真门 ⭐

这一步是"核心哲学"那条约束的落地机制——**在写任何 HTML 之前**，先定两件事：用什么模式转换、哪些内容允许动哪些不允许动。跳过这一步，模型很容易滑向"摘要型"，把规格里的字段/状态/异常当冗余删掉。

#### 1) 选转换模式（3 选 1，默认推荐双层型）

| 模式 | 适用 | 处理已有内容的方式 |
|---|---|---|
| **保真型** | 正式 PRD、研发规格、合同、操作手册 | 逐章保留，只重组呈现，不摘要 |
| **双层型（默认推荐）** | 大多数产品/技术文档 | 主线精简（Hero/架构/主线章节）+ 完整规格折叠保留 |
| **摘要型** | 汇报、路演、管理层快速评审 | 允许大幅压缩，但必须说明删了什么 |

**判定流程**：如果用户没明说，默认**双层型**。不能由模型自行默认摘要型——摘要型是用户显式要求"给高管看 / 控制在 N 页"时才用。文档是要被研发/运营拿去落地的（含字段、状态、异常、验收标准），一律不进摘要型。

#### 2) 内容保真门（写 HTML 前给源章节打标签）

通读完源文档后、动手设计 HTML 前，先把每个源章节归到下面 6 类之一。这一步决定"这段内容允许怎么处理"：

| 标签 | 处理方式 | 例子 |
|---|---|---|
| **核心需求** | 100% 保留，展开 + 视觉强化 | 产品定位、核心场景、MVP 必做项 |
| **决策依据** | 保留（可移位到决策点旁），不摘要 | "为什么选这群人"的 5 个理由 |
| **执行规格** | 保留（可折叠），**折叠保留全文不摘要** | 字段定义、状态机、错误码、验收标准 |
| **重复表达** | 合并到唯一主定义处，其他位置用锚点引用 | "产品定位"在概览和详述里各写一遍 |
| **历史说明** | 进附录 | 变更日志、已废弃方案 |
| **可删除** | **必须说明删除理由**，否则不删 | 与主题无关的离题内容 |

**一句话原则**：**去重 = 合并重复表达，不是删减不同层次的细节。** 概览与详细规格、主流程与数据模型、MVP 范围与验收标准——这些是**不同层次**，不是重复，都该保留。

详见 `references/fidelity-and-mapping.md`（含压缩预算启发式、语义重复 vs 必要重复对照、源章节映射模板）。

### Step 2 · 检查逻辑完整性（最重要）

逐章问自己：**这章删了，逻辑链会不会断？**

如果会断，就是核心章节，不能省略，不能折叠。
如果用户给的 markdown 缺了关键环节（比如产品文档没有数据来源、技术方案没有错误处理），**必须主动指出并补上**——不能以"原文没写"或"这是技术细节"为由跳过。

典型容易遗漏的关键环节：
- 产品文档：**数据来源**、关键约束的依据、未决项
- 技术方案：**错误处理**、回滚策略、监控告警
- API 文档：**鉴权**、限流、错误码

详见 `references/completeness-check.md`。

### Step 3 · 设计信息分层（6 类手段，不只是折叠）

**关键认知**：信息分层 ≠ 折叠。分层是更大的概念——决定"用户何时、如何看到什么信息"。折叠只是其中一种手段。

有 6 类分层手段，按需组合：

| 手段 | 适用 | 例子 |
|---|---|---|
| **空间位置** | 跨章节复用的约束、术语 | 关键约束侧栏卡片、章节边注 |
| **视觉权重** | 突出关键，弱化次要 | 关键数字大字号 vs 普通文字（字号+色+字重叠加） |
| **Hover 悬浮** | 短信息按需查看，不破坏主线 | `<abbr>` 术语定义、引用摘要、错误码含义 |
| **折叠展开** | 整段内容按需查看 | 决策溯源、详细配置、附录（详见下表） |
| **跳转链接** | 关联但分散 | "详见 X.Y"、跨文档引用 |
| **附录归档** | 几乎不看的元数据 | 变更日志、归档历史 |

**判断流程**（按"用户何时需要"）：

| 用户场景 | 推荐手段 |
|---|---|
| 始终需要看 | 空间位置（主区）+ 视觉权重（突出） |
| 主线阅读时偶尔需要 | **Hover 悬浮**（不破坏主线） |
| 主线已表达，深入时需要 | 折叠 + 强化引导 |
| 关联章节的关联点 | 跳转链接 |
| 跨章节通用约束 | 侧栏固定卡片 |
| 偶尔查阅的参考 | 附录 + 强化引导 |
| 几乎不看 | 附录 + 朴素折叠 |

**Hover vs 折叠的边界**：
- Hover 适合**短信息**（术语、定义、引用摘要），阅读成本极低，但**移动端不可用**
- 折叠适合**长内容**（多段、列表、表格），需要点击但移动端友好
- **关键信息不能只藏在 hover 里**（移动端、截图、SEO 都会丢）

详见 `references/layering-patterns.md`。

#### 折叠的三类内容（手段 4 的细化）

当判定用"折叠"时，再细分三类：

| 类型 | 判定 | 呈现 |
|---|---|---|
| **核心** | 删了逻辑链会断 / 决策依据 / Phase 1 必做 | **完全展开** + 视觉强化（不该折叠） |
| **重要但按需查阅** | 主线已表达，深入细节 | 折叠 + **强化引导**（色条 + badge + 计数） |
| **真次要** | 几乎不看 | 朴素折叠 |

**关键**：折叠 ≠ 次要。"为了简洁而折叠"是常见错误——如果内容是核心架构或决策依据，应该展开。

详见 `references/folding-decision.md`。

### Step 4 · 选表现形式（两层决策）

**核心原则**：不要直接按"内容类型"选形式——**先问"读者在这里需要做什么"，再问"内容是什么类型"**。

内容驱动会陷入"系统架构 = Mermaid graph"的机械映射；读者驱动会让你意识到：同一份架构图，给新人看用 Mermaid，给专家查接口用表格，给高管汇报用 Hero 数字。

#### 第一层：读者目的驱动（顶层决策）

读者阅读时有 6 类目的，每类有最优形式：

| 读者目的 | 推荐形式 | 多场景例子 |
|---|---|---|
| **快速判断**（要不要继续读） | 关键数字 + 状态色 + Hero 卡片 | 产品北极星 / 技术 SLO / API 健康状态 / 操作手册的关键阈值 |
| **理解关系**（这跟那有什么关联） | Mermaid graph / 并排卡片 | 产品双渠道架构 / 技术微服务拓扑 / API 字段依赖 |
| **做决策**（选 A 还是 B） | 对比表格 + callout + 状态色 | 产品人群选择 / 技术选型 / API 多版本取舍 |
| **执行操作**（按步骤做） | 时间轴 / 序号步骤卡片 | 产品 onboarding / 技术部署流程 / 操作手册故障排查 |
| **查阅参数**（找到具体值） | 表格 + hover tooltip | 产品能力矩阵 / 技术配置项 / API 字段定义 / 错误码 |
| **学习概念**（理解新术语） | hover 定义 + 折叠详解 | 产品业务术语 / 技术缩写 / API 协议概念 |

#### 第二层：内容类型默认映射 + 何时别用

判定读者目的后，参考下表选具体形式。**每行附"何时别用"避免机械套用**：

| 内容类型 | 默认形式 | ✅ 何时该用 | ❌ 何时别用 |
|---|---|---|---|
| 关键数字 | 大字号卡片 + tabular-nums | 阅读时需要被锚定（北极星、红线、规模） | 工程师精确查阅（用表格更高效） |
| 核心约束 / 警告 | callout（左 4px 色条 + 浅色背景） | 前置到章节开头，扫读时必看到 | 长论述（用独立章节展开） |
| 系统架构 | Mermaid graph + subgraph + classDef | 多组件复杂关系 | 简单 1-2 组件（文字更清楚） |
| 业务流程 | Mermaid flowchart + 决策分支色分 | 多步骤含分支 | 简单 1-2 步（文字 + 列表更高效） |
| 时序交互 | Mermaid sequenceDiagram + Note | 多角色跨系统调用 | 单角色单系统（用流程图） |
| 状态转换 | Mermaid stateDiagram-v2 | 状态 ≥ 3 且转换复杂 | 状态 ≤ 2（用表格） |
| 阶段演进 | Mermaid timeline | 时间序列清晰、阶段多 | 单一时间点（用文字） |
| 多方案对比 | 并排卡片 + 不同色边框 | 2-3 方案核心对比 | > 4 方案（用表格更紧凑） |
| 长列表 | 响应式表格 + sticky header + 斑马线 | 同维度多对象对比 | 异构内容（用嵌套卡片） |
| 术语 / 缩写 | `<abbr>` hover 或 tooltip | 主线阅读时偶尔需要定义 | 关键术语（用折叠详解更完整） |
| 详细配置 | 表格 + hover tooltip | 按需查阅具体值 | 决策驱动场景（先突出关键差异） |

#### 第三层：视觉节制（选完形式后的约束）

选好形式之后，还要约束**用多少**。卡片、图表、callout 都是好的，但堆多了视觉权重会被稀释——所有元素同权 = 没有重点。这是选形式时的第三道闸，和"何时别用"并列：

- **同层连续卡片建议 ≤ 6 张**：超过 6 张，读者的扫视注意力抓不住重点，考虑改用表格或分组
- **普通列表不全部卡片化**：只有"需要并列对比/各自有独立结论"的才卡片化；纯列举用列表更高效
- **每章最多一个主视觉重点**：一张主架构图 或 一个关键数字区，不要一章里并排 3 张同等大小的图
- **核心内容不因"页面整齐"全部同权展示**：不要为了让卡片网格对齐，把核心架构和次要配置做成一样大小的卡片

**为什么有这条**：本 skill 的一个对照案例里，一份保真型转换产出 121 张卡片——结果视觉重点完全稀释，读者扫完记不住任何一个。卡片数量不是越多越好，**克制的视觉权重才能突出真正的重点**。

#### 决策流程图

```
读者要做什么？
├── 快速判断     → 关键数字 / 状态色 / Hero
├── 理解关系     → Mermaid graph / 并排卡片
├── 做决策       → 对比表格 + callout
├── 执行操作     → 时间轴 / 步骤
├── 查阅参数     → 表格 + hover
└── 学习概念     → hover 定义

确认目的后，再问"内容是什么类型"
→ 查上面的默认映射 → 选具体形式
→ 检查"何时别用"列，避免机械套用
→ 最后过"视觉节制"：用多了吗？有重点吗？
```

详见 `references/visualization-patterns.md`。

### Step 5 · 系统扫描每章（写完后必做）

写完 HTML 后**逐章扫描**，对每章问：

1. **核心论点是什么？**（用一句话总结）
2. **当前设计是否让这论点成为视觉重点？**
3. **是否有信息平铺、缺少视觉层级？**

**不要只盯用户指出的单点**。如果用户指出一处问题，主动扫描全文找类似的——常见的同类问题：
- 表格里关键行淹没在普通行中
- 多卡片网格中所有卡片同等视觉权重（没突出关键）
- 数字藏在文字段落里（应该 关键数字卡片化）
- 折叠内容没有强化引导（朴素 summary，用户不会点开）

### Step 5.5 · 源章节映射 + 交付报告 ⭐

Step 5 是**质量审查**（每章核心论点突出吗），这一步是**完整度证明**（源文档的东西都在吗）。两者目的不同，不能合并。尤其是双层型/保真型转换，**交付时必须能自证没有无意遗漏**——否则很容易像摘要型一样丢内容却不自知。

#### 1) 源章节映射表

交付 HTML 前，生成一张 `源章节 → HTML 章节 → 处理方式` 的映射表（心里过一遍或写在交付说明里）：

| 源 Markdown 章节 | 对应 HTML 位置 | 处理方式 |
|---|---|---|
| 1. 定位 | Hero 区 | 完整展开 |
| 2-4. 用户/场景/模式 | §1 主线章节 | 合并呈现 |
| 5. 数据来源 | §2 | 完整展开 |
| 6-8. 字段/状态/异常 | §3 折叠规格层 | 折叠保留 |
| 附录 | §X | 附录保留 |

处理方式只能是 5 种之一：**完整展开 / 合并呈现 / 折叠保留 / 附录保留 / 删除并说明理由**。每一行源章节都必须出现在表里——没出现的 = 可能被无意删了。

#### 2) 交付报告（精简版，5 行）

交付 HTML 时附一份简短报告，让用户一眼看到"我动了什么"：

```
源 21 章 → HTML 11 节（合并 10 章，详见映射）
删除：无 / 或 删了 X（理由：…）
折叠保留的核心规格：6 处（字段/状态/异常/验收）
估算保留率：~90%（核心需求 100%）
```

详见 `references/fidelity-and-mapping.md`（含完整模板和 v15/v16 对照案例）。

## 视觉系统

### 状态色（4 态，全局一致）

| 状态 | 用途 | Tailwind 配色 |
|---|---|---|
| **success**（绿） | 可做 / 已完成 / 推荐项 | `text-success` `border-success` `bg-success` |
| **warning**（黄） | 部分支持 / 待决 / 警告 | `text-warning` `border-warning` `bg-warning` |
| **danger**（红） | 禁止 / 红线 / 失败 | `text-danger` `border-danger` `bg-danger` |
| **info**（蓝） | 信息 / 引用 / 提示 | `text-info` `border-info` `bg-info` |

每色三态：text（文字）/ border（边框）/ bg（背景，加 `/20` `/30` 控制透明度）

### 卡片设计

- 圆角 8px，subtle border (`border` 色)
- hover 时 `translateY(-1px)` + shadow 加深
- 关键卡片用 `border-2`（2px 加粗边框）
- 关键数字用 `text-3xl` 以上 + `font-bold` + `tabular-nums`

### Callout

左 4px 色条 + 浅色背景 + 1px 边框。用于：
- `info`（蓝）：补充说明、引用其他文档
- `success`（绿）：核心论点、最佳实践
- `warning`（黄）：注意事项、Phase 边界
- `danger`（红）：红线、必做约束、安全机制

### 折叠 details 强化引导

如果 details 包含"重要但按需查阅"的内容，summary 必须强化：

```html
<details class="card border-l-4 border-l-warning-border">
  <summary class="px-4 py-3 cursor-pointer font-medium flex items-center hover:bg-gray-50">
    <span class="summary-icon-closed mr-2 text-warning-text">▸</span>
    <span class="summary-icon-open mr-2 text-warning-text">▾</span>
    <span class="text-sm flex-1">标题</span>
    <span class="ml-auto flex items-center gap-2">
      <span class="badge bg-warning-bg text-warning text-[10px] border border-warning-border">🔒 决策依据</span>
      <span class="text-[10px] text-ink-500">4 个理由</span>
    </span>
  </summary>
  <!-- 内容 -->
</details>
```

四个强化元素：色条 / 图标颜色 / badge / 内容计数。

- [ ] **内容保真门**：源章节是否都归了 6 类标签？核心需求/决策依据/执行规格是否 100% 保留（执行规格可折叠但保留全文，不摘要）
- [ ] **逻辑链条无断点**：每章都是逻辑链上的环节，关键环节都补齐（数据来源、错误处理、回滚、监控等）
- [ ] **每章核心论点突出**：扫读时第一眼能看到核心信息，不是埋在段落中间
- [ ] **读者目的驱动选形式**：每个表现形式是否对应了读者目的（快速判断/理解关系/做决策/执行操作/查阅参数/学习概念），而不只是按"内容类型 → 形式"机械套用？
- [ ] **"何时别用"检查**：选定的形式是否避开了表格里的"何时别用"反例（如关键数字卡片用在工程师精确查阅场景就是错的）？
- [ ] **分层手段多元化**：不只依赖折叠——术语用 hover、跨章节约束用侧栏、关联内容用跳转链接（详见 `references/layering-patterns.md`）
- [ ] **hover 不藏关键信息**：hover 只用于"补充信息"，主线能独立阅读；考虑移动端访问场景
- [ ] **折叠合法性**：折叠的内容是"真次要"或"重要但按需查阅"，不是"为了简洁而藏起来的核心信息"
- [ ] **折叠引导**：所有折叠都有色条 + badge + 内容计数（真次要除外）

- [ ] **关键约束前置**：每章第一屏就能看到核心约束，不必读完才理解

- [ ] **TOC 同步**：侧边栏 TOC 链接和章节 id 一一对应

## 常见反模式（避免）

1. **"Markdown + CSS"伪重构**：只换字体颜色和加表格斑马线，没做空间重组 → 不算认知降维
2. **ASCII 图直接搬运**：架构图、流程图仍然用 ASCII 代码块 → 应该转 Mermaid 或嵌套卡片
3. **Tab 隐藏核心对比**：双人群/双方案对比用 Tab 切换，首屏看不到 → 应该并排卡片
4. **为了简洁折叠核心**：把"5 个理由"或"8 个模块"折叠让排版更清爽 → 应该展开
5. **朴素 summary**：折叠的 summary 只有标题，没色条没 badge → 用户不会点开
6. **数字藏在文字里**：关键数字 写在段落中间 → 应该 关键数字卡片大字号
7. **逻辑链断点**：跳过"数据来源"或"错误处理"等关键环节 → 逻辑不成立
8. **只盯用户指出的单点**：用户指出一处问题，不扫描全文找同类 → 系统性问题漏诊
9. **把"信息分层"等同于"折叠"**：6 类分层手段（空间/视觉/hover/折叠/跳转/附录）只用 1 类 → 该用 hover 的术语被折叠到附录，该用侧栏的约束被重复 N 次
10. **关键信息只藏在 hover 里**：移动端用户、截图、SEO 都拿不到 → hover 只用于"补充信息"，主线必须能独立阅读
11. **忽视移动端**：把核心提示或定义只放在 hover 里 → 手机用户完全看不到
12. **用特定场景词命名通用原则**（元反模式）：把"关键数字应该突出"写成"KPI 应该突出"——技术文档读者误以为"我没有 KPI 就不需要这种呈现"。同类陷阱：
    - "护城河要前置"（商业锚定）→ 应为"核心价值要前置"
    - "Phase 1 必做"（产品发版锚定）→ 应为"MVP 必做"
    - "反馈飞轮"（产品锚定）→ 应为"反馈循环"
    - "5 个理由要展开"（具体例子当通用原则）→ 应为"决策依据要展开"

    **识别 + 抽象方法**：见文首"适用文档类型 + 识别场景锚定"章节。原则用中性词，多场景配例子，场景词只下沉到具体例子里。
13. **机械套用"内容类型 → 形式"映射**（与元反模式 #12 同源）：看到"系统架构"就用 Mermaid graph，看到"关键数字"就用大字号卡片——不问读者目的、不查"何时别用"。同一份架构图给新人看用 Mermaid、给专家查接口用表格、给高管汇报用 Hero 数字。**先问读者要做什么，再选形式**。详见 Step 4 两层决策。
14. **混淆"语义重复"与"必要重复"**（本 skill 最严重的内容丢失模式）：看到两处讲同一件事就删一处——但很多"看似重复"其实是不同层次。判定：
    - **语义重复（真重复，合并）**：同一结论在概览和详述里各完整写一遍 → 保留一处主定义，其他位置改成锚点引用"详见 §X.Y"
    - **必要重复（不是重复，都保留）**：概览 vs 详细规格、主流程 vs 数据模型、MVP 范围 vs 验收标准、风险 vs 对应产品机制——这些是**不同读者、不同目的**下需要的不同表达，删了任一处都会让某类读者断链
    
    **一句区分法**：问"这两处是给同一类读者、回答同一个问题吗？"——是 → 合并；否（不同读者/不同问题）→ 都保留，可用锚点交叉引用。详见 `references/fidelity-and-mapping.md`。
